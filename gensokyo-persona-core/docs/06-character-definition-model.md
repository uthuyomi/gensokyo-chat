# 06. キャラクター定義モデル

## 6-1. 目的
キャラ定義を文章塊ではなく構造化データとして扱う

## 6-2. キャラ定義の4層

### A. Core Persona
- 一人称
- 二人称
- 性格軸
- 価値観
- 世界観上の立場
- 禁則

### B. Style Persona
- 語彙傾向
- 文長
- テンポ
- 比喩パターン
- 反応癖

### C. Behavior Rules
- 褒め方
- 励まし方
- 断り方
- 相談時の入り方
- 真剣な場面でのトーン

### D. Scene Modulation
- 通常
- 子ども相手
- 落ち込んだ相手
- SOS相手
- 技術質問相手
- 初対面
- 親密状態
- 距離がある相手

## 6-2a. 多言語対応のための第5層

### E. Locale Surface
- `ja-JP`
- `en-US`
- 将来の `zh-*`, `ko-KR` など

ここでは以下を持つ
- 一人称 / 二人称の locale 別表現
- 語尾 / formality / directness
- 子ども向けの言い換え方
- SOS時のその言語として自然な支え方
- 比喩や語彙の locale 別表層

## 6-3. 保存形式

### 候補
- YAML
- JSON
- Pydantic モデル

### 推奨
- 永続ファイルは YAML
- runtime では Pydantic へロード

## 6-4. 重要方針
- 「相談モード」「子ども向けモード」という汎用モードを持つのではなく、
  **そのキャラの相談時挙動 / 子ども対応 / SOS時対応** を持つ
- 場面ごとに変わるのはキャラ性ではなく、キャラ本人の振る舞い方
- control plane は英語ベースで統一し、Locale Surface で実際の発話言語を切り替える
