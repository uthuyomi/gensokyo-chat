# 04. 設計原則

## 4-1. 人格の正本は backend に置く
- UI には見た目情報だけ残す
- 人格 / 口調 / few-shot / 制約 / 安全上書きは backend 管理

## 4-1a. UI は人格制御をしない
- UI はキャラ prompt を組み立てない
- UI は口調調整をしない
- UI は相談 / SOS / 子ども向けの分岐をしない
- UI は generation params を人格目的で操作しない
- UI は「どのキャラで返すか」と「入力文脈」を渡すだけにする

## 4-2. 「キャラクター」と「会話方針」を分離する
- キャラクター = 不変の核
- 会話方針 = 一般モードではなく、そのキャラが状況ごとにどう振る舞うかを選ぶ層

## 4-2a. 「制御言語」と「発話言語」を分離する
- runtime の control plane は英語ベースで持つ
- 日本語・英語など実際の発話面は locale ごとの style pack で持つ
- 英語化するのは制約・優先順位・安全・場面制御であって、キャラ味そのものを平板化することではない

## 4-3. 「意味生成」と「キャラ表現」を分離する
- 何を返すか
- どう言うか

を別層にする

## 4-4. 常にキャラ本人として生成する
- 子ども向けでもキャラ性を下げない
- 相談でも汎用相談AIに寄せない
- SOSでも無個性テンプレに差し替えない

## 4-5. プロンプト本数ではなくキャラ魂 + 場面別振る舞い部品で管理する
- 巨大な prompt バリエーション集にしない
- 固定人格 + キャラ固有の場面別振る舞い + 安全制約で組み立てる
- さらに fixed English control blocks + locale style blocks で組み立てる

## 4-6. 速度モードを正式な機能として持つ
- `fast`
- `balanced`
- `deep`

を runtime レベルで持つ
